## What this iteration adds

Pipeline v2 extends pipeline v1 with three new feature groups:

- **Log windows**: activity split into 10-day and 30-day slices, plus their ratio (recent vs older behavior). The intuition is that a user who listened heavily two months ago but stopped recently is a different risk profile than someone consistently inactive.
- **Refined completion metrics**: `completion_ratio` replaces `completion_rate` from v1, plus `avg_secs_per_active_day` and usage standard deviation.
- **Interaction flags**: four binary features combining payment and activity signals (e.g. `low_usage_no_autorenew`, `recent_activity_recent_txn`).

| Feature group                                    | Count  |
| ------------------------------------------------ | ------ |
| From pipeline v1 (baseline + transaction + logs) | 26     |
| New log window features                          | 15     |
| Interaction flags                                | 4      |
| **Total**                                        | **44** |

## Model comparison

All numbers are from the validation set: 80/20 stratified split, seed=42, ~194k users.

| Model                             | ROC AUC | Avg Precision | Precision@0.5 | Recall@0.5 | F1@0.5 |
| --------------------------------- | ------- | ------------- | ------------- | ---------- | ------ |
| LightGBM                          | 0.7793  | 0.4760        | 0.2866        | 0.5757     | 0.3827 |
| XGBoost                           | 0.7791  | 0.4760        | 0.2858        | 0.5757     | 0.3820 |
| HistGradientBoosting              | 0.7786  | 0.4735        | 0.2820        | 0.5791     | 0.3793 |
| GradientBoosting (200k subsample) | 0.7722  | 0.4643        | 0.7775        | 0.2700     | 0.4008 |
| Random Forest (depth=15, n=200)   | 0.7675  | 0.4534        | 0.2954        | 0.5559     | 0.3858 |

LightGBM and XGBoost are functionally identical in AUC and average precision. LightGBM was chosen for the final notebook — it trains faster on the full 776k training rows and the early stopping callback is simpler to configure.

Note on GradientBoosting: it was trained on a 200k stratified subsample because sklearn's classic GBC is prohibitively slow at full scale. Its AUC is lower but it still validates the same direction as the other boosting models.

**Final config (notebook 08):**

```python
LGBMClassifier(
    n_estimators=1000,
    learning_rate=0.05,
    max_depth=8,
    num_leaves=31,
    class_weight="balanced",
    random_state=42,
)
```

Early stopping with patience=50 on validation log-loss. All 1,000 trees were used — logloss kept improving throughout, never triggering the stopping criterion. Final validation AUC: **0.7778**.

## Ranking metrics (Recall@K)

Instead of just evaluating at a fixed threshold, the model is evaluated as a ranking tool: sort users by predicted probability and measure what fraction of churners fall into the top K.

| Budget  | % of val users | Churners captured |
| ------- | -------------- | ----------------- |
| 5,000   | 2.6%           | 23.8%             |
| 10,000  | 5.1%           | 36.1%             |
| 25,000  | 12.9%          | 51.1%             |
| 50,000  | 25.7%          | 64.1%             |
| 100,000 | 51.5%          | 80.2%             |

A random list of the same size would capture roughly K/194k of churners — around 2.6% for K=5,000. The model is about 9× better than random in the high-precision region.

## What doesn't work / limitations

- **Single split**: all metrics come from one holdout. The numbers are reasonable but there's no cross-validation, so confidence intervals are unknown.
- **No uplift**: a high churn probability doesn't mean the user will respond to retention. Some high-risk users will churn regardless; some low-risk users might respond if contacted. Separating the two requires a different modeling approach.
- **Competition labels**: `is_churn` is defined by the Kaggle competition, not by a real business. In production the definition might differ (e.g., 60-day window, different plan types).
- **No drift monitoring**: the model is a snapshot trained on 2017 data. It would need retraining to stay relevant over time.
