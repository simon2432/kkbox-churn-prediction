## Raw data and leakage guardrails

| File                  | Role                                                                           |
| --------------------- | ------------------------------------------------------------------------------ |
| `train_v2.csv`        | `msno`, `is_churn` (label)                                                     |
| `members_v3.csv`      | Registration date, `bd`, `gender`, geography fields                            |
| `transactions_v2.csv` | Subscription rows: dates, plan, payment, `is_auto_renew`, `is_cancel`          |
| `user_logs_v2.csv`    | Daily listening rows (March 2017 window used for log aggregates in this track) |

**Transaction / member cutoff:** **`2017-02-28`**. Only rows with `transaction_date <= cutoff` enter transaction aggregates. That aligns subscription-side features with the churn horizon and avoids bleeding in later renewal events as ŌĆ£known past.ŌĆØ

**User logs:** Aggregates are built from **`user_logs_v2.csv`** (listening in the label-relevant window ŌĆö March 2017 in the notebooks; same source file as in `pipeline_v1`). They are merged **after** the baseline + transaction-derived frame is built.

## Data handling (preprocessing, not feature engineering)

Applied consistently in the path that became **`pipeline_v1`** (and explored stepwise in notebooks 02ŌĆō03):

- **CSV loading:** `pandas.read_csv` (default encoding in the local environment).
- **Dates:** Columns stored as `YYYYMMDD` strings are converted with `to_datetime(..., format="%Y%m%d", errors="coerce")` ŌåÆ invalid values become **NaT**.
- **After merges:** Missing transaction counts / cancel fields for users without pre-cutoff history are filled with **0**; `has_txn_history` encodes whether any transaction existed.
- **Members:** `gender` missing ŌåÆ **`"unknown"`**. **`bd`** outside **[0, 100]** ŌåÆ treated as invalid (**NA**).
- **Logs alignment:** Users with no log rows get log feature columns filled with **0** when aligned to the train user list.
- **Final modelling matrix:** All **26** numeric features get **`fillna(0)`** before training / CSV export.
- **No** explicit **encoding** of categoricals in the exported artefact (e.g. `gender` is cleaned in the wide table but **not** among the 26 features).
- **No** **scaling** in the pipeline; tree models handle mixed scales; logistic regression in early notebooks used raw features (limitation noted below).

## Feature engineering (three layers)

### Layer A ŌĆö Baseline transaction + member (9 features)

From **`train_v2`** Ōł¬ **`members_v3`** Ōł¬ pre-cutoff **`transactions_v2`**:

| Feature                                                                                        | Idea                               |
| ---------------------------------------------------------------------------------------------- | ---------------------------------- |
| `txn_count`, `has_txn_history`                                                                 | Volume / existence of paid history |
| `auto_renew_last`, `payment_plan_days_last`, `plan_list_price_last`, `actual_amount_paid_last` | Last transaction snapshot          |
| `cancel_count`, `has_cancelled`                                                                | Cancellation behaviour             |
| `days_since_registration`                                                                      | Tenure vs cutoff                   |

Medians used later for v2 (`payment_plan_days_median`, `actual_amount_paid_median`) are computed in the aggregation step but surfaced when building v2 fields.

### Layer B ŌĆö Transaction extensions (6 features)

Same cutoff and aggregates; derived **after** merge (as in notebook 05 / `add_v2_v3_derived_features` in code):

| Feature                                 | Idea                                           |
| --------------------------------------- | ---------------------------------------------- |
| `days_since_last_txn`, `has_recent_txn` | Recency vs renewal                             |
| `txn_per_day`                           | Intensity between first and last txn           |
| `avg_payment`                           | Typical payment (median-based imputation path) |
| `price_discount`                        | Listed vs paid, clipped at 0                   |
| `plan_consistency`                      | Last plan length vs typical                    |

### Layer C ŌĆö User log aggregates (11 features)

Per-user stats from daily log rows (March window in notebooks / full file read in pipeline with same definitions):

| Feature                                      | Idea                                                                                       |
| -------------------------------------------- | ------------------------------------------------------------------------------------------ |
| `total_secs_sum`, `total_secs_mean`          | Listening time                                                                             |
| `active_days`                                | Days with any play                                                                         |
| `num_100_sum`, `num_unq_sum`, `num_unq_mean` | Completion-ish counts, unique tracks                                                       |
| `completion_rate`                            | Mean-day completion proxy (v1 / notebooks; see pipeline v2 for `completion_ratio` variant) |
| `days_since_last_log`, `has_recent_activity` | Log recency                                                                                |
| `peak_day_secs`, `activity_cv`               | Peaks and day-to-day variability                                                           |

**Total for pipeline v1 / notebook 06 frame:** **9 + 6 + 11 = 26** features (+ `msno`, `is_churn`).

## Models tried (through notebook 06)

All nonŌĆōcross-validated results below use a **single 80/20 train/validation split**, stratified on `is_churn`, **`random_state=42`**, unless noted.

| Stage       | Models                                                                                  | Best validation ROC AUC (typical) | Notes                                                                                    |
| ----------- | --------------------------------------------------------------------------------------- | --------------------------------- | ---------------------------------------------------------------------------------------- |
| Notebook 04 | Logistic regression (9 features)                                                        | **~0.567**                        | Default `sklearn`; no `class_weight`; convergence issues possible with expanded numerics |
| Notebook 05 | LR (v2 features), DT (`max_depth=5`), RF (`n=100`, `depth=10`, balanced)                | **~0.673** (RF)                   | Recency (`days_since_last_txn`) dominates RF importance                                  |
| Notebook 06 | RF (v3 features, deeper RF), **LightGBM** (500 trees, lr 0.05, depth 6, early stopping) | **~0.776** (LightGBM)             | Log block lifts AUC strongly; `days_since_last_log` leads importance in RF view          |

**Thresholds:** For the v2 RF, lowering the decision threshold (e.g. **0.4**) traded precision for very high recall on churners. For **LightGBM** in notebook 06, the doc-shaped trade-off is similar: higher recall (~0.68ŌĆō0.91) at thresholds **0.3ŌĆō0.4** with lower precision ŌĆö business cost drives the cutoff.

## Results snapshot (from notebook tables)

- **Baseline LR (notebook 04):** accuracy inflated by majority class; churn recall at 0.5 very low (~0.06); AUC ~0.567.
- **v2 RF (notebook 05):** AUC ~0.673; balanced weights materially increase minority-class sensitivity vs LR.
- **v3 LightGBM (notebook 06):** AUC ~**0.776**; at 0.5 threshold, churn recall ~**0.57** and precision ~**0.28** in the documented run ŌĆö both improved vs v2 RF at 0.5.
- **Feature importance (RF on v3):** log-related columns sum to a large share of split importance (~54% in the documented deeper RF); **`days_since_last_log`** at the top.

## Limitations (still true for this v1 track)

1. **Single holdout split** ŌĆö metrics vary if the split changes; no k-fold CV in the notebooks.
2. **Leaderboard gap** ŌĆö strong competition solutions sit higher (~0.89ŌĆō0.91 AUC); this repo is a structured learning path, not a full submission stack.
3. **Log window** ŌĆö aggregates over the log calendar window; no sequence models or per-session features in v1.
4. **Linear model fairness** ŌĆö LR without scaling underperforms when v2 numerics are added; trees are the intended workhorse from v2 onward.
5. **Pipeline v2** (optional) adds windows, interaction-style flags, and refined completion metrics ŌĆö **outside** the frozen 26-feature v1 contract documented here.
